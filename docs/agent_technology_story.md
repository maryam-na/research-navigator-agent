# Agent Technology Story

ResearchNavigator Agent demonstrates a local-first research discovery agent: a bounded workflow that plans, calls tools, checks evidence, enforces safety policy, and produces a grounded research brief for a focused local paper corpus.

## Why An Agent Is Useful For Research Discovery

Research discovery is not a single summarization step. A careful reviewer needs to ingest papers, separate evidence from interpretation, connect related claims, find limitations, inspect citations, form possible gaps, and check whether generated ideas are overreaching. An agent is useful because it can make that sequence explicit and inspectable.

ResearchNavigator turns the workflow into a tool trajectory. The user sees which local tool is responsible for each stage, what data it consumes, what safety gate applies, and what evidence IDs support the final result.

## Agent Loop

The current deterministic loop is:

```text
plan
  -> ingest
  -> extract
  -> retrieve evidence
  -> build graph
  -> discover gaps
  -> generate hypotheses
  -> evaluate
  -> produce grounded brief
```

In the Streamlit app, the `Pipeline Trace` tab exposes this trajectory through the ADK-facing manifest, planned tool sequence, policy gates, MCP tool list, generated artifacts, and reproducible local commands.

## Tools Used

The agent-facing tool layer lives in `app/adk_tools.py`.

Primary tools:

- `run_local_demo_pipeline`: runs the full local pipeline.
- `ingest_local_papers`: parses local PDFs, chunks text, extracts statements, and writes SQLite records.
- `build_local_graph`: builds the NetworkX graph from local statements.
- `discover_local_research_gaps`: generates evidence-backed gaps, speculative hypotheses, and experiment plans.
- `evaluate_local_outputs`: checks grounding, safety, traceability, and testability.
- `search_local_corpus`: retrieves from local processed artifacts.
- `inspect_evidence`: returns compact evidence rows for statement IDs.
- `generate_local_research_brief`: exports a grounded Markdown brief.
- `check_local_policy`: validates proposed tool actions against local policy rules.
- `describe_agent_capabilities`: exposes the capstone concept story.
- `planned_tool_trajectory`: shows the deterministic sequence before answering.

Supporting deterministic modules:

- `tools/pdf_tools.py`
- `tools/extraction_tools.py`
- `tools/storage_tools.py`
- `tools/graph_tools.py`
- `tools/gap_tools.py`
- `tools/safety_tools.py`

- `tools/evaluation_tools.py`

## Deterministic Baseline

The local baseline is intentionally deterministic:

- SQLite stores structured records.
- NetworkX builds the graph.
- Rule-based extraction identifies research statement types.
- Deterministic gap and hypothesis generation require local evidence IDs.
- Evaluation checks score grounding, safety, testability, and traceability.
- The no-secrets scan is local and pattern based.
- The agent trace is regenerated by `scripts/export_agent_trace.py`.

This makes the demo reproducible for judges and keeps privacy boundaries clear.

## Future Optional Gemini Layer

A future Gemini-assisted version could help with ranking, semantic grouping, richer statement extraction, and clearer natural-language explanations. That extension should be optional and gated by:

- explicit user consent,
- no secret commits,
- local policy checks,
- prompt-injection handling,
- evidence ID requirements,
- safety/evaluation regression cases,
- human review before using generated research ideas.

The deterministic local baseline does not call Gemini, does not require API keys, and does not train or fine-tune models.

## Why Local-First Matters

Research papers can include restricted, private, or unpublished material. A local-first workflow helps users review a focused corpus without uploading documents to a hosted service. It also makes judging easier: the project can be run from a cloned repository with local commands instead of a paid deployment.

Local-first boundaries:

- papers remain under `data/papers/`;
- processed outputs stay under `data/processed/`;
- generated demo artifacts stay under `data/generated/`;
- no cloud deployment is required;
- no external web search is part of the local baseline;
- no API key is needed for the deterministic demo.

## How ADK Fits

`app/agent.py` defines a Google ADK-facing root agent when ADK is available. `app/adk_tools.py` exposes the deterministic tool functions and the inspectable tool manifest. The ADK wrapper demonstrates the core agent pattern without forcing a live LLM dependency into the local demo.

Judges can inspect:

- `app/agent.py`
- `app/adk_tools.py`
- `Pipeline Trace` in Streamlit
- `data/generated/agent_trace_demo.json`

## How MCP Fits

`app/mcp_server.py` exposes selected deterministic tools through a local MCP-compatible server. This lets MCP clients inspect and call the same bounded local capabilities without changing the trust model.

Run it locally:

```bash
make mcp
```

The MCP layer is not a cloud deployment. It is a local wrapper around the same evidence, policy, and evaluation boundaries.

## Safety Boundaries

- Paper text is treated as untrusted data.
- Prompt injection is flagged and not followed.
- Gaps and hypotheses must cite local statement IDs.
- Hypotheses are labeled `speculative_research_hypothesis`.
- Fake citations and unsupported claims are flagged.
- Cloud deployment, external browsing, email, training, fine-tuning, and unsafe writes are blocked in the local baseline policy.
